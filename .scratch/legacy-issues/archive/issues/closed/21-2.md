 Comprehensive Fix for Role Validation Issues - Datastar Way

 Core Issues Identified

 1. Game starts with more players than roles - validation isn't preventing game start
 2. Leaderless toggle doesn't update button state - signals not properly updating
 3. Role assignment ignores UI configuration - preset auto-scaling happens regardless of display

 Root Cause Analysis

 The implementation violates Datastar's core principle: "The DOM is the state". We're trying to manage complex validation state through signals when we should be  │
 returning updated DOM fragments.

 Proper Datastar Solution

 Phase 1: Fix Role Assignment Logic

 File: /workspace/nix/app/internal/game/roles.go

 1. Fix AssignRolesWithConfig to respect actual configuration:
 // For custom configs, use exact counts from RoleTypes
 // For presets, only auto-scale if room.RoleConfig.AutoScaleEnabled is true
 2. Add AutoScaleEnabled field to RoleConfiguration struct

 Phase 2: Fix Checkbox Toggle Updates

 File: /workspace/nix/app/internal/handlers/role_config.go

 1. UpdateLeaderlessGame handler should:
   - Update the room configuration
   - Return a DOM fragment that includes:
       - The updated role configuration component
     - The start button with correct disabled state
   - NOT try to manage this through signals

 Example:
 // Re-render affected components
 component := components.StartGameSection(room, validationState)
 html := renderToString(component)
 sse.MergeFragments(html, datastar.WithSelector("#start-game-section"))

 Phase 3: Simplify Validation Display

 File: /workspace/nix/app/internal/views/pages/lobby.templ

 1. Remove complex signal management for validation
 2. Create a StartGameSection component that includes:
   - Start button with proper disabled state baked into HTML
   - Validation message if any
   - All state determined server-side

 templ StartGameSection(room *game.Room, validation ValidationState) {
     <div id="start-game-section">
         if validation.CanStart {
             <button class="btn btn-primary" data-on-click="...">
                 Start Game
             </button>
         } else {
             <button class="btn btn-primary" disabled>
                 Start Game
             </button>
             <div class="alert alert-warning mt-2">
                 { validation.ValidationMessage }
             </div>
         }
     </div>
 }

 Phase 4: Fix SSE Updates

 File: /workspace/nix/app/internal/handlers/sse.go

 1. When role config changes, send updated DOM fragments:
   - Role configuration component
   - Start game section with validation
 2. Stop trying to manage validation through signals

 Phase 5: Fix StartGame Handler

 File: /workspace/nix/app/internal/handlers/actions.go

 Keep the SSE fragment approach but ensure validation actually prevents game start:
 1. Check validation state
 2. If invalid, return error fragment
 3. If valid, proceed with game start

 Key Principles to Follow

 1. DOM is the state - Return updated HTML, don't try to manage complex state in signals
 2. Signals are for temporary UI state - Like "isLoading" during a request
 3. Server determines all state - Validation happens server-side, HTML reflects result
 4. Always return 200 OK - Use fragments for errors

 Implementation Order

 1. Fix role assignment to respect configuration (most critical bug)
 2. Update handlers to return proper DOM fragments
 3. Simplify templates to remove complex signal logic
 4. Test all scenarios

 This approach:
 - Follows Datastar's philosophy
 - Simplifies the code significantly
 - Ensures validation actually works
 - Makes the UI reactive without complex client-side state
