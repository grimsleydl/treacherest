When testing via running `dev` then opening a browser and starting a game, then opening that game in multiple browsers


If I start the game, the countdown appears on one browser but not the others. They load their role assignment successfully but only after refreshing.


One series of thoughts and affected files that may be related to this change:

Here is a clean list of the files that are directly and tangentially related to the data flow for fixing this bug.
Core Logic & State
nix/app/internal/handlers/actions.go
Reason: Contains the StartGame handler, which is the entry point for this flow, and the runCountdown goroutine. This is where the initial state change occurs.
nix/app/internal/handlers/sse.go
Reason: This file contains the bug. Specifically, the StreamLobby function incorrectly handles game-related events, causing other players' connections to drop. The StreamGame function is also relevant as it's the target SSE stream for redirected clients.
nix/app/internal/handlers/handlers.go
Reason: Defines the Handler struct, the EventBus (the central pub/sub mechanism), and the Event struct. It's the core of the event-driven pattern.
nix/app/internal/game/room.go
Reason: Defines the Room struct and the GameState constants (StateLobby, StateCountdown, StatePlaying), which are critical to the logic in the handlers.
nix/app/internal/game/roles.go
Reason: The AssignRoles function is called directly from StartGame.
UI Templates & Routing
nix/app/internal/views/pages/lobby.templ
Reason: Defines the "Start Game" button and the LobbyBody component that is updated by the StreamLobby SSE handler.
nix/app/internal/views/pages/game.templ
Reason: This is the destination page for all players after the game starts. Its GameBody is what should display the countdown for everyone.
nix/app/cmd/server/server.go
Reason: Sets up the Chi router, mapping the /room/{code}/start POST request to the StartGame handler and the SSE endpoints.
Supporting Backend Files
nix/app/internal/store/memory.go
Reason: Provides the in-memory storage for Room objects, which is accessed and updated throughout the "start game" process.
nix/app/internal/game/player.go
Reason: Defines the Player struct, which is fundamental to the game state.
Testing & Debugging
nix/app/internal/handlers/actions_test.go
Reason: Contains tests for the StartGame handler.
nix/app/internal/handlers/sse_test.go
Reason: Contains unit and integration tests for the SSE streaming logic.
nix/app/internal/handlers/sse_multibrowser_test.go
Reason: Highly relevant. This test file is specifically designed to simulate and verify behavior across multiple concurrent browser connections, which is the exact scenario where the bug occurs.
issues/open/3.md
Reason: The original bug report describing the user-facing issue.
