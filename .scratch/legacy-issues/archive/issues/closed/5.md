# Issue #5: Browser SSE Connection Exhaustion During Game Countdown

## Summary
During game countdown, the browser creates dozens of pending SSE connections to the same endpoint 

## Discovery Context
While investigating Issue #4 (navigation deadlock after game start), user reported that the original navigation issue persisted even after implementing the EventBus actor model fix. Upon requesting screenshots, discovered browser DevTools showing 20+ pending SSE connections all to the same room endpoint.

## Root Cause Analysis

### The Loop
1. User starts game → countdown begins (5 seconds)
2. Game page loads with `data-on-load="@get('/sse/game/ROOM')"` 
3. SSE connection established to `/sse/game/ROOM`
4. Countdown timer publishes `countdown_update` event every second
5. **BUG**: StreamLobby handler treats `countdown_update` as a game transition event and sends redirect: `window.location.href = '/game/ROOM'`
6. But user is ALREADY on the game page!
7. This causes page to reload, creating a NEW SSE connection
8. Steps 3-7 repeat every second during the 5-second countdown
9. Result: Dozens of pending SSE connections accumulate

### Why This Breaks Navigation
- Browsers have a limit on concurrent connections per origin (typically 6-8)
- With 20+ pending SSE connections, the browser's connection pool is exhausted
- New requests (like navigating to create a new room) cannot establish connections
- This manifests as "endless loading" when trying to navigate

## Attempted Fixes

### 1. EventBus Actor Model (Partially Successful)
- Refactored EventBus from mutex-based to actor model
- Eliminated the Subscribe/Unsubscribe deadlock
- However, this didn't solve the SSE connection exhaustion issue

### 2. StreamLobby Event Handling (Attempted)
Modified StreamLobby to not redirect on `countdown_update` events:
```go
case "game_started":
    // Redirect to game page when game starts
    sse.ExecuteScript("window.location.href = '/game/" + roomCode + "'")
    return
case "countdown_update", "game_playing":
    // These events happen after game has started
    // Players should already be on the game page, so just close this lobby connection
    return
```

## Files Involved

### Core Files
- `nix/app/internal/handlers/sse.go` - StreamLobby handler with problematic redirect logic
- `nix/app/internal/handlers/actions.go` - runCountdown publishes events every second
- `nix/app/internal/views/pages/game.templ` - Game page with SSE trigger on load

### Event Flow
1. `StartGame` (actions.go) → starts countdown goroutine
2. `runCountdown` (actions.go) → publishes `countdown_update` every second
3. `StreamLobby` (sse.go) → receives event and redirects
4. `GamePage` (game.templ) → reloads with new SSE connection

## Solution Implemented

### Root Cause (Revised)
The issue was not in StreamLobby redirect logic, but in how Datastar handles `data-on-load` during DOM morphing:
1. `data-on-load` was placed directly on the `#game-container` element
2. Every countdown update morphed this element
3. Datastar re-evaluated `data-on-load` on each morph, creating a new SSE connection
4. This happened 6-7 times per second value (5,4,3,2,1,0), resulting in ~36 connections

### The Fix
Following the pattern from lobby.templ, we separated the SSE trigger from morphable content:

1. **Template Structure** (game.templ):
```templ
templ GameBody(room *game.Room, currentPlayer *game.Player) {
    // data-on-load is on wrapper div that never gets morphed
    <div data-on-load={ "@get('/sse/game/" + room.Code + "')" }>
        @GameContent(room, currentPlayer)
    </div>
}

templ GameContent(room *game.Room, currentPlayer *game.Player) {
    <div id="game-container" class="container" data-signals={...}>
        // morphable content here
    </div>
}
```

2. **SSE Handler Update** (sse.go):
```go
func (h *Handler) renderGame(sse *datastar.ServerSentEventGenerator, room *game.Room, player *game.Player) {
    // Use GameContent (without wrapper) to prevent re-triggering data-on-load
    component := pages.GameContent(room, player)
    html := renderToString(component)
    sse.MergeFragments(html,
        datastar.WithSelector("#game-container"),
        datastar.WithMergeMode(datastar.FragmentMergeModeMorph))
}
```

### Verification
- Before fix: ~36 SSE connections created during 5-second countdown
- After fix: Only 1 SSE connection (as expected)
- Navigation issues resolved: Can navigate back and create new rooms
- No connection pool exhaustion

## Status: FIXED

The fix has been implemented and tested successfully. Both the SSE connection exhaustion and the navigation issues have been resolved.
