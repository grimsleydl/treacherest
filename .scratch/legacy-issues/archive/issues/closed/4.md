Navigating back/trying to join new rooms in the same tab after a game has started doesn't seem to work.


Even with just a single player, this flow results in a seemingly endless (or at least quite long) "loading spinner" in the browser:

- Navigate to main page
- Create a game
- Start the game
- Navigate "back" in the browser twice (to get back to the main page)

## Status: FIXED

This issue was caused by SSE connection exhaustion (see Issue #5). The browser's connection pool was exhausted by dozens of pending SSE connections created during game countdown, preventing new navigation requests from completing.

### Solution
Fixed by preventing SSE connection re-creation during DOM morphing. See Issue #5 for detailed fix information.

### Verification
- Can now navigate back after starting a game
- Can create new rooms without loading spinner
- No deadlock or connection exhaustion occurs



Core Logic & State Management
nix/app/internal/handlers/handlers.go
Reason: Defines the EventBus and its sync.RWMutex, which is the root cause of the deadlock. The Subscribe and Unsubscribe methods are the key participants in the race condition.
nix/app/internal/handlers/sse.go
Reason: Contains the StreamLobby and StreamGame handlers. These are the two goroutines that execute the Subscribe/Unsubscribe pattern, leading to the deadlock when a user navigates between them.
nix/app/internal/game/room.go
Reason: Defines the GameState (StateLobby, StatePlaying) which controls the application's flow and determines which SSE stream a user should be connected to.
nix/app/internal/store/memory.go
Reason: Manages the lifecycle and state of all Room objects, which are accessed and modified by all relevant handlers.
Action & Page Handlers
nix/app/internal/handlers/actions.go
Reason: The StartGame function triggers the game_started event that causes all lobby SSE connections to close and redirect. The LeaveRoom function is also part of the user navigation flow.
nix/app/internal/handlers/pages.go
Reason: The JoinRoom and GamePage handlers are responsible for rendering the pages that initiate the SSE connections via data-on-load attributes. The logic within JoinRoom handles how a player enters a new lobby after leaving a previous game.
UI Templates (Client-Side Triggers)
nix/app/internal/views/pages/lobby.templ
Reason: Its data-on-load attribute in LobbyBody initiates the StreamLobby SSE connection.
nix/app/internal/views/pages/game.templ
Reason: Its data-on-load attribute in GameBody initiates the StreamGame SSE connection. The cancellation of this connection upon back-navigation is the first step in the bug sequence.
nix/app/internal/views/pages/home.templ
Reason: The entry point for creating or joining a new room after navigating back.
nix/app/internal/views/layouts/base.templ
Reason: Includes the datastar.js script responsible for all client-side SSE and DOM morphing logic.
Server Configuration & Routing
nix/app/cmd/server/server.go
Reason: Defines the application's routes, wiring URLs like /sse/lobby/{code} to the StreamLobby handler.
nix/app/cmd/server/main.go
Reason: The application's main entry point. The IdleTimeout: 0 setting is relevant to how SSE connections are managed by the server.
