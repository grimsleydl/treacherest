# Issue 15: UI Testing Strategy - Prevent JavaScript Runtime Errors

## Problem Statement

While implementing the leaderless games feature (issue #9), we introduced JavaScript errors in the UI that were not caught by our test suite. Specifically, multi-line JavaScript expressions in Datastar's `data-on-change` attributes caused runtime parsing errors that completely broke the UI.

## Current Testing Gaps

### 1. No UI/Template Testing
- Templ templates are compiled but not tested for runtime behavior
- JavaScript expressions in data attributes are not validated
- No tests for Datastar-specific syntax and patterns

### 2. No Browser Testing
- Can't catch JavaScript runtime errors
- Can't verify DOM manipulation works correctly
- Can't test SSE updates and real-time features

### 3. No Integration Testing
- Backend tests don't verify the full request/response cycle
- No tests for the interaction between Go handlers and Templ templates
- SSE event handling is untested

## Proposed Solutions

### 1. Add Playwright Tests (High Priority)
As mentioned in CLAUDE.md, we should use Playwright MCP for browser testing:
- Test all UI interactions before committing
- Verify no JavaScript errors in console
- Test SSE updates work correctly
- Test form submissions and validations

Example test:
```javascript
test('role configuration toggles work without errors', async ({ page }) => {
  // Navigate to lobby
  await page.goto('/room/TEST1');
  
  // Check for console errors
  const errors = [];
  page.on('console', msg => {
    if (msg.type() === 'error') errors.push(msg.text());
  });
  
  // Toggle leaderless game option
  await page.click('#allow-leaderless');
  
  // Verify no errors
  expect(errors).toHaveLength(0);
  
  // Verify confirmation dialog appears
  await expect(page.locator('text=Enable leaderless games?')).toBeVisible();
});
```

### 2. Add Templ Expression Validation (Medium Priority)
Create a linter or test that validates JavaScript expressions in templates:
- Check for syntax errors in data-* attributes
- Verify Datastar action syntax (@post, @get, etc.)
- Ensure expressions are single-line where required

### 3. Add Template Rendering Tests (Medium Priority)
Test that templates render without errors:
```go
func TestRoleConfigurationRendering(t *testing.T) {
    room := &game.Room{
        RoleConfig: &game.RoleConfiguration{
            AllowLeaderlessGame: true,
        },
    }
    
    component := components.RoleConfiguration(room, cfg, roles)
    
    // Render to string and check for issues
    html := renderToString(component)
    
    // Verify no multi-line expressions in data attributes
    assert.NotContains(t, html, "data-on-change=\"\n")
}
```

### 4. Add Datastar-Specific Guidelines (High Priority)
Document patterns that work and don't work with Datastar:

**DO:**
```html
data-on-change="condition && @post('/url')"
data-on-change="condition ? @post('/url') : null"
```

**DON'T:**
```html
data-on-change="
  if (condition) {
    @post('/url')
  }
"
```

### 5. Pre-commit Hooks (Low Priority)
- Run `templ generate` and check for errors
- Run a basic syntax check on generated HTML
- Verify no console.error calls in development

## Implementation Steps

1. **Immediate**: Set up Playwright with a few critical path tests
2. **Short-term**: Document Datastar patterns and anti-patterns
3. **Medium-term**: Add template rendering tests
4. **Long-term**: Full UI test coverage for all features

## Success Criteria

- No JavaScript runtime errors make it to production
- All UI interactions are tested before merge
- Template changes are validated for correct syntax
- Datastar expressions follow documented patterns
- Browser console remains error-free during normal usage

## Notes

This issue highlights a critical gap - we have good backend test coverage but zero frontend/UI testing. Given that this is a web application with real-time features, browser testing is essential.

The key lesson: **If it runs in the browser, it needs browser tests.**
