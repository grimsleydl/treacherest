# Issue #23: API Improvements and RESTful Endpoint Design

## Overview

This issue focuses on improving the backend API design to be more RESTful and maintainable. Currently, many endpoints use complex signal-based updates that could be simplified with better endpoint structure.

## Background

During the implementation of Issue #21 (role validation), it was identified that the current role configuration endpoints could be more RESTful and easier to maintain. The existing approach works but has some areas for improvement in terms of API design.

## Improvements Needed

### 1. Simplified Role Count Endpoint
**Priority: MEDIUM**

**Current Implementation:**
- Role count updates go through generic handlers with complex signal parsing
- Data flows through multiple layers of signal management
- Harder to test and maintain

**Proposed Improvement:**
- Create dedicated RESTful endpoints for role count updates
- Cleaner separation of concerns
- Easier to test and debug

**Implementation:**
```go
// New endpoint structure
POST /room/{code}/config/role-type/{roleType}/count
{
  "count": 3
}

// Or even simpler:
POST /room/{code}/config/role-type/{roleType}/increment
POST /room/{code}/config/role-type/{roleType}/decrement
```

**Benefits:**
- More RESTful API design
- Clearer intent in endpoint naming
- Easier to implement rate limiting per action
- Better error handling per specific operation
- Simplified testing

### 2. Consistent Response Patterns
**Priority: LOW**

**Current State:**
- Mix of SSE fragment returns and JSON responses
- Some endpoints return different formats

**Proposed Improvement:**
- Standardize response patterns across all role configuration endpoints
- Consistent error response format
- Better HTTP status code usage

### 3. API Documentation
**Priority: LOW**

**Current State:**
- API endpoints are not formally documented
- Developers need to read code to understand endpoints

**Proposed Improvement:**
- Add OpenAPI/Swagger documentation for all endpoints
- Include request/response examples
- Document expected behavior

## Implementation Plan

### Phase 1: New Role Count Endpoints
1. **Add new routes** in `/workspace/nix/app/cmd/server/main.go`:
   ```go
   r.Post("/room/{code}/config/role-type/{roleType}/count", h.UpdateRoleTypeCount)
   r.Post("/room/{code}/config/role-type/{roleType}/increment", h.IncrementRoleCount)  
   r.Post("/room/{code}/config/role-type/{roleType}/decrement", h.DecrementRoleCount)
   ```

2. **Implement handlers** in `/workspace/nix/app/internal/handlers/role_config.go`:
   - Extract role type and room code from URL parameters
   - Simple JSON body parsing (no complex signals)
   - Update room configuration
   - Return standardized SSE fragments

3. **Update frontend** to use new endpoints:
   - Modify role configuration component
   - Use simpler data-on-click attributes
   - Remove complex signal management where possible

### Phase 2: Response Standardization
1. **Create response helper functions**:
   - Standard error response format
   - Consistent SSE fragment wrapping
   - Proper HTTP status codes

2. **Update existing endpoints** to use new patterns:
   - Consistent error handling
   - Standardized success responses
   - Better logging

### Phase 3: Documentation
1. **Add API documentation**:
   - Document all role configuration endpoints
   - Include examples
   - Add to project documentation
